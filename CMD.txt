--build makes sure all images are rebuilt. Next time, without code changes, you can omit --build.


# View logs of one specific service
docker compose logs -f discordbot
docker compose logs -f autoredeemgifts
docker compose logs -f mysql

# Stop everything (data stays preserved in volume)
docker compose down

# Stop everything AND wipe the database (fresh start)
docker compose down -v

# Rebuild only one container after a code change
docker compose up --build discordbot

# If you also want the data gone:
docker compose down -v

# The built images always remain (they take up space). Remove those separately:
docker compose down --rmi all

# Or everything at once (containers + volumes + images):
docker compose down -v --rmi all


Ik heb de docker-compose.yml, nginx.conf en API-config bekeken om het concreet te maken. Hier is de checklist voor Hetzner:

1. VPS voorbereiden
Hetzner Cloud server aanmaken (CX22 volstaat ruim voor deze stack), Ubuntu 24.04 LTS
SSH-key toevoegen, root-login en password-auth uitzetten, non-root sudo-user aanmaken
Hetzner Cloud Firewall instellen: alleen 22 (SSH), 80, 443 open naar buiten
Op de server zelf ook ufw configureren als tweede laag, fail2ban voor SSH
Docker Engine + docker compose plugin installeren
2. DNS & domein
Domein/subdomein A-record naar het VPS-IP wijzen (voor de frontend; evt. een tweede subdomein als je de API los wil aanspreken)
3. TLS — ontbreekt nu nog
Je mrbloop_FE/nginx.conf luistert alleen op poort 80, zonder TLS. Voeg een edge-reverse-proxy toe die HTTPS regelt, bv. Caddy (automatisch Let's Encrypt) vóór de frontend service, of breid nginx uit met certbot. Caddy is het minste werk: 1 extra service in compose die naar frontend:80 proxyt.

4. Secrets opnieuw aanmaken op de server
.env/db.env staan terecht in .gitignore, dus die moet je los aanmaken op de VPS (niet clonen):

db.env met nieuw, sterk wachtwoord (niet hergebruiken van lokaal)
mrbloop_BE/discordbot/.env — productie DISCORD_TOKEN
mrbloop_BE/autoredeemgifts_v5/.env — nieuwe JWT_SECRET, REDEEM_SECRET, MOD_PASSWORD, en ALLOWED_ORIGINS naar je echte domein i.p.v. localhost
mrbloop_FE/.env.production staat al goed (VITE_API_URL=/api)
5. Compose hardenen voor productie
Verwijder of bind mysql poort 3307:3306 niet naar 0.0.0.0 — niet nodig in productie, en anders je DB publiek bereikbaar. Zo nodig SSH-tunnel gebruiken voor Workbench-toegang.
autoredeemgifts poort 8000 hoeft niet publiek; nginx proxyt /api/ al intern via het docker-netwerk. Bind 'm desnoods naar 127.0.0.1:8000:8000 voor debug, niet naar buiten.
Let op: de poller draait in de FastAPI lifespan van autoredeemgifts. Niet horizontaal schalen (geen 2e replica/worker), anders redeem't hij codes dubbel.
6. Bekende gotcha: rate limiting achter de proxy
api/limiter.py gebruikt get_remote_address (slowapi), wat het IP van de proxy ziet i.p.v. het echte client-IP zodra je achter Caddy + de nginx-proxy zit. Los op met --proxy-headers --forwarded-allow-ips op uvicorn of Starlette's ProxyHeadersMiddleware, anders is de rate limit straks effectief gedeeld door alle bezoekers (of omzeild).

7. Deployen

git clone <repo> && cd mrbloop_bot
# .env / db.env handmatig aanmaken
docker compose up -d --build
docker compose ps
docker compose logs -f
DB-migraties draaien automatisch via docker-entrypoint-initdb.d — werkt alleen op een lege mysql_data volume, dus dit is eenmalig bij eerste opstart.
8. Na deploy testen
Frontend bereikbaar over HTTPS, login werkt, /api/ calls slagen
Discord bot online in de juiste server, slash commands gesynced
Poller-logs tonen daadwerkelijke code-ophaal/redeem-cycli
/docs van FastAPI is publiek — overweeg dit af te schermen of te accepteren dat het read-only informatief is
9. Operationeel
Log rotation instellen voor docker (json-file driver groeit anders onbeperkt) — max-size/max-file in compose of /etc/docker/daemon.json
Geautomatiseerde mysqldump-backup (cron) van mysql_data, liefst off-server (Hetzner Storage Box / S3)
Uptime-check op frontend + API health endpoint (bv. UptimeRobot)
Update-procedure documenteren: git pull && docker compose up -d --build, plus handmatig migraties draaien bij schema-wijzigingen (init-scripts draaien niet opnieuw op een bestaand volume)
